# asr-bench v0.4 Scoping — NVIDIA NeMo / Canary-Qwen

> **Status:** Scoping / research document. Surfaces decisions and open questions to feed a later interactive brainstorming + design-spec session. Choices below are *recommendations*, not commitments.
> **Date:** 2026-06-07. **Target:** add an NVIDIA NeMo engine (Canary-Qwen family) as asr-bench's 4th `Engine` family, "local engines only" through v0.4.

## Executive summary

NVIDIA NeMo is the natural v0.4 engine: **Canary-Qwen-2.5B** currently tops the HuggingFace Open ASR Leaderboard at **5.63% mean WER** (vs Whisper Large-V3 around ~7.4–7.8% on the same board), and asr-bench is exactly the tool to produce a controlled head-to-head on real lecture audio. The dominant constraint is the same wall WhisperX hit: NeMo is a pure-Python `py3-none-any` wheel, but it depends on PyTorch, and **PyTorch ships no CUDA wheels for Python 3.14 (cp314)** — so NeMo cannot do GPU inference in asr-bench's 3.14 core. The proven answer is to reuse the `.venv-whisperx` pattern: a dedicated **Python 3.12 `./.venv-nemo`** driven by a new **`nemo_runner.py`** subprocess that emits pure JSON on stdout. NeMo is heavier and more dependency-pinned than WhisperX (numpy>=2.0 / transformers / lightning collisions), so it must get its **own** venv — not the core, not `.venv-whisperx`. Recommended scope: add **two** models — `canary-qwen-2.5b` (best-in-class English WER, but **no native timestamps**, 40s encoder window) and **`parakeet-tdt-0.6b-v2`** (native word/segment timestamps, ~2GB VRAM, ~3380 RTFx, the VTT-friendly workhorse) — and **defer the `engines/` package split** to a separate refactor PR after NeMo is live-validated.

## Models in scope

Primary requested model is Canary-Qwen-2.5B; Parakeet-TDT-0.6B-v2 is the strongly-recommended companion because Canary-Qwen has no native timestamps and asr-bench's pipeline assumes timed cues.

| Model | Params | License | Native timestamps? | English mean WER (Open ASR LB) | Speed | VRAM | API surface |
|---|---|---|---|---|---|---|---|
| **Canary-Qwen-2.5B** *(primary)* | 2.5B (FastConformer enc from canary-1b-flash + Qwen3-1.7B dec via LoRA) | CC-BY-4.0, **not HF-gated** | **No** (text only; needs forced alignment for VTT) | **5.63%** (#1 at launch) | 418 RTFx; 40s encoder window → needs chunked inference | Not officially published; ~5–6GB fp16 weights est. — **measure on 5090** | `speechlm2.models.SALM` `.generate(prompts=[...], max_new_tokens=...)` |
| **Parakeet-TDT-0.6B-v2** *(workhorse)* | 600M (FastConformer + TDT) | CC-BY-4.0, not gated | **Yes** (word/segment/char) | 6.05% | ~3380–3386 RTFx @ batch 128; up to **24 min single pass** | ~2GB | `nemo_asr.models.ASRModel` `.transcribe(timestamps=True)` |
| Canary-1B-Flash *(defer v0.5)* | 883M (FastConformer + Transformer) | CC-BY-4.0 | Experimental (en/de/fr/es) | — (encoder source for Canary-Qwen) | >1000 RTFx | — | standard `ASRModel` |
| Parakeet-TDT-0.6B-v3 *(defer v0.5)* | 600M, 25 EU langs + auto LID | **NVIDIA Open Model License** (not CC-BY-4.0) | Yes (word) | — | — | — | standard `ASRModel` |
| *(context) Whisper Large-V3* | 1550M | MIT | Yes (segment) | ~7.4–7.8% | baseline | ~1.2GB (this corpus) | faster-whisper |

Notes that bite the integration:
- **Canary-Qwen output is clean punctuated text — excellent for WER, but exposes NO word/segment timestamps.** Its encoder was trained on a 40s max window (1024 max tokens), so a ~50-min lecture **must** use NeMo chunked inference (`chunk_len_in_secs=40.0`, non-overlapping). The example `max_new_tokens=128` is far too small for a 40s chunk and will truncate transcripts — must be raised substantially per chunk.
- **Parakeet-v2 is the timestamp source of truth:** `out[0].timestamp['segment']` (start/end seconds + text) maps straight into WebVTT cues; `['word']` feeds the `_Words_<Model>.json` sidecar exactly like WhisperX already produces.
- Per-dataset Canary-Qwen WER: LS-clean 1.60, LS-other 3.10, AMI 10.18, GigaSpeech 9.41, Earnings22 10.42, SPGISpeech 1.90, TED-LIUM 2.72, VoxPopuli 5.66 (short-clip leaderboard numbers — chunked long-form may differ, see open questions).

## Install & environment

**Hard blocker (same as WhisperX):** PyTorch publishes only CPU wheels for cp314 (pytorch/pytorch #169929 closed not-planned; #156856 tracks 3.14). asr-bench's core runs on Python 3.14 → **NeMo GPU inference is impossible in-core; a subprocess to a ≤3.13 venv is mandatory, not optional.**

- **Package:** `nemo_toolkit[asr]` (latest 2.7.3, 2026-04-23, `python_requires>=3.10`). Pure-Python `py3-none-any` wheel — all version pressure comes from torch.
- **Python:** NeMo recommends/tests **3.12** (`conda create -n nemo python=3.12`). Use 3.12 for `./.venv-nemo`.
- **PyTorch:** NeMo needs torch **2.6+** (nightly docs say 2.7+). Canary-Qwen-2.5B needs **NeMo ≥ 2.5.0** (for `speechlm2.models.SALM`).
- **CUDA / Blackwell:** RTX 5090 is sm_120 → install **cu128** torch FIRST, then NeMo. This is the documented WhisperX trap: a bare `pip install nemo_toolkit[asr]` (or whisperx) can pull **CPU-only torch** on Windows. Always verify `torch.cuda.is_available() == True` in the venv. NeMo also offers `[cu12]`/`[cu13]` extras (CUDA 13 recommended for Canary-Qwen, 12.9 reported working), but for the 5090 the cu128 torch-first order is the safe path.
- **Footprint:** NeMo + cu128 torch + transformers + lightning + audio deps ≈ **6–10+ GB on disk**, long install.
- **Known conflict risks:** numpy>=2.0 incompatibility (NeMo #14505), transformers/huggingface_hub pins (#10940), Lightning coupling. **These are a second hard reason not to share a venv with WhisperX.** Also: torch 2.6+ defaults `torch.load(weights_only=True)` — some checkpoints need `TORCH_FORCE_NO_WEIGHTS_ONLY_LOAD=1` in the subprocess env.
- **Ecosystem flux:** NVIDIA split NeMo (~Dec 2025) — repo now centers on Speech (ASR/TTS/Speech-LLM); first post-split Speech release scheduled **June 2026**. Newest Canary features sometimes require git-trunk NeMo over the PyPI release. Pin a known-good `(Python, torch+cu, nemo)` triple for reproducibility.

**Recommended venv strategy:** dedicated `./.venv-nemo` (Python 3.12), provisioned by a new `setup_nemo_venv.ps1` mirroring `setup_whisperx_venv.ps1`:
1. create py3.12 venv → 2. `pip install torch torchvision --index-url https://download.pytorch.org/whl/cu128` **FIRST** → 3. `pip install nemo_toolkit[asr]` → 4. verify `torch.cuda.is_available()` True **and** a tiny Canary/Parakeet load. Record the exact validated triple in CLAUDE.md/SPEC.md afterward (as was done for WhisperX: torch 2.8.0+cu128 / whisperx 3.8.6 / pyannote 4.0.4).

## Engine integration

The integration is **overwhelmingly additive** and mirrors `WhisperXEngine` almost exactly. Concrete anchors in `asr_bench.py`:

- **Engine ABC** (`asr_bench.py:816-823`): `class Engine(ABC)` with `name: str` + one abstract `run(self, entry, pairs, cfg) -> ModelResult`. `NeMoEngine` subclasses it.
- **ENGINES registry** (`:1236-1239`, extended at `:1479` via `ENGINES["whisperx"] = WhisperXEngine`). Add `ENGINES["nemo"] = NeMoEngine` the same late-append way.
- **MODELS registry** (`:114-160`): add `canary-qwen-2.5b` and `parakeet-tdt-0.6b-v2` entries with `"engine": "nemo"` + a `"nemo_model"` key (e.g. `"nvidia/canary-qwen-2.5b"`), mirroring how faster-whisper uses `fw_name` and nim uses `riva_model`. Add `_MODEL_VRAM_COST` entries once VRAM is measured.
- **Dispatch loop** (`main()`, `:2793-2800`): `engine_cls = ENGINES.get(entry["engine"]); engine_cls().run(...)` is fully generic — **zero changes** once registered.
- **Optional ad-hoc ids:** a `nemo:<model>` regex in `resolve_model_entry` (`:167-213`) mirroring `_NIM_ADHOC_RE` (`:162`) — only if ad-hoc NeMo ids are wanted (open question).

**In-process vs subprocess — recommend SUBPROCESS as the default and primary path.** Justification: (1) torch has no 3.14 wheels and core runs on 3.14 → in-process is impossible in the normal install; (2) NeMo is heavier/more CUDA-pinned than WhisperX → benefits from its own isolated venv; (3) subprocess gives clean per-model VRAM/process teardown across the dispatch loop (importing the massive NeMo+LLM stack in-process risks VRAM/state contamination between sequential models). Mirror the WhisperX adapter trio:
- `NeMoAdapter` ABC + `SubprocessNeMo` + `FakeNeMoAdapter` (+ optional `InProcessNeMo` gated on `importlib.util.find_spec("nemo")`/`"torch"`, exactly like `make_whisperx_adapter` at `:1388-1400`).
- `make_nemo_adapter(cfg)` auto-selector with `_default_nemo_python()` probing `./.venv-nemo/Scripts/python.exe` and `./.venv-nemo/bin/python` (mirrors `_default_whisperx_python` at `:1378-1385`).
- `SubprocessNeMo.transcribe` mirrors `SubprocessWhisperX` (`:1365-1375`): build `cmd=[exe, *runner_args]`, pass secrets via **env not argv**, `subprocess.run(capture_output=True, text=True, timeout=..., check=False)`, raise on nonzero with `proc.stderr[:500]`, else `NeMoResult.from_dict(json.loads(proc.stdout))`. `_RUNNER_PATH` resolved relative to `__file__`.

**`nemo_runner.py` JSON contract** (new standalone sibling, the 2nd broken-out runner). Mirror `whisperx_runner.py` (`:1-183`): keep ALL torch/nemo imports inside functions; `build_arg_parser` (`--audio --model --device --language` + chunk flags); and a `main()` that emits **exactly ONE JSON document on stdout** while routing all library chatter to stderr. **This dual redirect is non-negotiable** — NeMo/Lightning/HF are noisy on stdout; `whisperx_runner.py:159-177` does `os.dup2(2,1)` + `sys.stdout=sys.stderr` during processing and restores fd 1 only to write the final `json.dumps(out)`. Proposed output shape:

```json
{
  "text": "<full transcript>",
  "segments": [{"start": 0.0, "end": 4.2, "text": "..."}],
  "words":    [{"word": "...", "start": 0.0, "end": 0.3}],
  "transcribe_sec": 12.34,
  "vram_peak_bytes": 5912345678,
  "language": "en"
}
```
Add a `NeMoResult` dataclass with `from_dict()` + `text()` + a to-cues helper (mirror `WhisperXResult` at `:1243-1270`). For RTFx, **time the `model.transcribe()`/`.generate()` call inside the runner** and return `transcribe_sec` (more accurate than wall-clocking the whole subprocess, which includes model load) — the engine can still wall-clock as a fallback like `WhisperXEngine` does at `:1433`/`:1440`. For VRAM, the runner can report `torch.cuda.max_memory_allocated()`, or the engine can reuse the in-process `VramSampler` (`:827-860`).

**Metrics infra reused UNCHANGED** (no new ClipResult/ModelResult fields, so **`schema_version` stays 1** and `render_markdown` + the JSON sidecar stay byte-for-byte stable): `ClipResult` (`:676-707`), `ModelResult` (`:710-788`) with all aggregates (avg_wer/mer/wil/cer, aggregate_rtfx, median_rtfx, median_sec_per_audio_min, peak_vram_bytes, hallucination_rate), `compute_word_metrics`/`normalize_for_wer` (`:328-430`), `compute_hallucination_signals` (`:402-404`), `write_whisper_vtt` (`:607-628`) + `group_words_into_cues` (`:895-917`), `render_markdown` (`:2132-2432`), and the sidecar writers (`_clip_to_dict :2027`, `_model_to_dict :2048`, `build_results_document :2082`, `write_results_json :2119`). `NeMoEngine.run` follows the `WhisperXEngine.run` body (`:1428-1473`) verbatim except the adapter call: `load_reference_text`/`detect_reference_origin` → `normalize_for_wer` → `compute_word_metrics` → `write_whisper_vtt` → `compute_hallucination_signals` → populate `ClipResult`/`ModelResult` with `engine="nemo"`.

**Thread a `nemo_python` venv path through** (mirrors `whisperx_python` at `:809`): add the field to `RunConfig` (`:791-813`), a `--nemo-python` CLI flag, wire into the `cfg=RunConfig(...)` build (`main()`, `:2775-2791`), add it to `_config_to_dict` (`:2069-2079`, keep it non-secret), and add a pre-flight venv/availability check near the WhisperX pre-flight (`:2711-2717`).

**Captions story for Canary-Qwen — decide explicitly (decision-log item):** because it has no native timestamps, either (a) **score WER-only, emit no VTT** (mark text-only in the report like NIM's `n/a` disk handling) — the lower-risk v0.4 scope — or (b) generate cues via forced alignment, reusing the WhisperX align stage on Canary-Qwen text. Parakeet-v2 sidesteps this entirely with native timestamps.

**Don't forget:** `test_engine_contract.py:27` hard-asserts `ENGINES.keys() == {'faster-whisper','nim','whisperx'}` — this **must** be updated to include `'nemo'` regardless of any package-split decision.

## The engines/ package split decision

**Recommendation: KEEP DEFERRING the split. Add `NeMoEngine` in-file for v0.4, then do the split as a separate, dedicated refactor PR after NeMo is live-validated and merged.**

The split was explicitly gated (CLAUDE.md 2026-06-04 entry / SPEC.md:178) on "a 4th engine family where the weight of three Engine subclasses becomes unwieldy" — and NeMo is that family, so the gate *is* met. But the rationale for still deferring:
- **Two unrelated risk sources should not be bundled.** v0.4's real risk is NeMo's heavy CUDA-pinned venv bridge (a repeat of the WhisperX subprocess saga, which forced three live-run fixes: pure-JSON stdout, token kwarg, default model). Folding a 2,849-line structural refactor into that makes regressions impossible to localize.
- **Large import surface.** The 231-test suite reaches **~60 symbols by flat name `asr_bench.X`** — including private `_repeat_coverage`, `_compression_ratio`, `_json_sanitize`, `_clip_to_dict`, `_model_to_dict`, `_reproducibility_command` — plus `from asr_bench import Cue/RunConfig/Pair/WhisperXResult`. A naive move breaks dozens of tests; the split needs an exhaustive re-export shim.
- **Circular-import hazard.** Engines depend on shared infra AND the registry maps names→engine classes. Requires clean layering (a torch-free `_core`/base that imports neither engines nor CLI) or you get a cycle. This also risks the `test_core_dependencies_are_minimal_and_torch_free` invariant if a split accidentally imports torch at module load.
- **Worst-possible packaging timing.** v0.4 is the pip-install + PyInstaller milestone (SPEC.md:152-153). Switching `pyproject.toml` from flat `py-modules=['asr_bench','asr_compare','whisperx_runner']` to packages changes setuptools discovery, console-entry resolution, and **how the runner siblings are located** (`SubprocessWhisperX` resolves runners relative to the module file — `:1329-1342`, `:1378`). Minimize moving parts in the packaging release.
- **Pure churn, no user-visible benefit.** Maintainability is real but should not ride shotgun on a feature release faculty IT actually cares about.

**Proposed layout WHEN the split is eventually done** (separate PR, acceptance gate = full 231-test suite green + `python asr_bench.py --help` + `pip install -e .` smoke):
```
asr_bench.py            # facade: shebang, main(), re-export shim, ENGINES assembly
asr_bench_core.py       # torch-free shared infra: Cue/WordMetrics/Pair/ClipResult/
                        #   ModelResult/RunConfig, Engine ABC, MODELS, resolve_model_entry,
                        #   metrics, normalize, VRAM/NVML, decode_to_pcm16,
                        #   VTT/JSON/markdown rendering, fusion + LLM backends
engines/__init__.py
engines/base.py         # re-export Engine + registry helper
engines/faster_whisper.py
engines/nim.py          # + NIM helpers
engines/whisperx.py     # + adapters + WhisperXResult
engines/nemo.py         # + adapters + NeMoResult
whisperx_runner.py      # TOP-LEVEL sibling (keep out of the package)
nemo_runner.py          # TOP-LEVEL sibling (keep out of the package)
```
Hard invariants the shim must preserve: `python asr_bench.py` still runs; `import asr_bench` still works; **every** one of the ~60 test-reached symbols (including private `_` functions) stays importable as `asr_bench.X`; `main()` stays in `asr_bench.py` so console script `asr_bench:main` is byte-stable. (Secondary future extraction target, out of scope here: fusion + LLM backends `run_fusion_stage`/`OllamaBackend`/`CliBackend`, ~400 lines.)

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| **Python 3.14 GPU hard blocker** — no torch CUDA wheels for cp314 | Subprocess to `./.venv-nemo` (Python 3.12) is mandatory; do not attempt in-core GPU. |
| **CPU-only torch trap on Windows** (bit WhisperX) | `setup_nemo_venv.ps1` installs cu128 torch FIRST; verify `cuda True` before declaring the venv ready. |
| **Dependency collisions** (numpy>=2.0 #14505, transformers/hf #10940, lightning) | Dedicated venv; do NOT share with core or `.venv-whisperx`. Pin a known-good triple. |
| **Canary-Qwen has no native timestamps** — breaks the "every engine yields timed cues" assumption | Decision-log item: WER-only/no-VTT (low risk) vs reuse WhisperX forced alignment. Ship Parakeet-v2 as the timestamp-capable model. |
| **40s encoder window + truncation** | Implement NeMo chunked inference (40s non-overlapping); raise `max_new_tokens` per chunk; concatenate before scoring. Validate on a real ~50-min lecture. |
| **Two divergent APIs** (`SALM.generate` vs `ASRModel.transcribe`) | `nemo_runner.py` branches on model class; budget for two code paths in one runner. |
| **Pure-JSON stdout fragility** (NeMo/Lightning/HF noisy) | Replicate `whisperx_runner.py` dual redirect (`os.dup2(2,1)` + `sys.stdout=sys.stderr`) verbatim. |
| **`weights_only` checkpoint-load failures** (torch 2.6+) | Set `TORCH_FORCE_NO_WEIGHTS_ONLY_LOAD=1` in the subprocess env. |
| **Undocumented VRAM** for Canary-Qwen | Measure live on the 5090 via NVML before merge; populate `_MODEL_VRAM_COST` from measurement. |
| **LLM-decoder hallucination** on noisy/OOD lecture audio | asr-bench's existing reference-free hallucination signals (repeat_coverage, compression_ratio) already flag this — a plus; watch on real audio. |
| **Parakeet-v3 license** (NVIDIA Open Model License, not CC-BY-4.0) | Defer v3 to v0.5; if added, document the distinct license. v2 is CC-BY-4.0. |
| **Windows friction** — NeMo's heavy paths are Linux-first | Live-validate on the Windows 11 / RTX 5090 reference box; if a dep forces WSL, document it (open question). |
| **`config_to_dict` omission** | Add `nemo_python` to `_config_to_dict` so the reproducibility command/sidecar isn't silently incomplete; keep any secret out. |
| **"Local engines only through v0.4" rule** | Default `NeMoEngine` to a local in-venv checkpoint; do NOT default to a hosted NeMo NIM/HTTP path. |

## Recommended approach & rough task breakdown

Mirror the WhisperX shipping precedent: build in-file, live-validate on the 5090, *then* merge; do the package split separately afterward.

1. **Pin the version triple** — install-test on Windows 11 / RTX 5090: find a `(Python 3.12, torch+cu128, nemo_toolkit)` combo that loads both `canary-qwen-2.5b` (via `SALM`) and `parakeet-tdt-0.6b-v2` (via `ASRModel`, `timestamps=True`) without resolver conflicts. Record it.
2. **`setup_nemo_venv.ps1`** — mirror `setup_whisperx_venv.ps1`: create `.venv-nemo`, cu128 torch FIRST, then `nemo_toolkit[asr]`, verify `cuda True` + tiny model load.
3. **`nemo_runner.py`** — standalone sibling: lazy heavy imports, `build_arg_parser`, branch on model class (SALM chunked-generate vs ASRModel.transcribe-with-timestamps), pure-JSON-stdout main with dual redirect, `TORCH_FORCE_NO_WEIGHTS_ONLY_LOAD=1`, emit the JSON contract above.
4. **`NeMoEngine` + adapters in `asr_bench.py`** — `NeMoResult` dataclass; `NeMoAdapter`/`SubprocessNeMo`/`FakeNeMoAdapter` (+ optional `InProcessNeMo`); `make_nemo_adapter` + `_default_nemo_python`; `NeMoEngine.run` cloned from `WhisperXEngine.run`.
5. **Register** — MODELS entries (`canary-qwen-2.5b`, `parakeet-tdt-0.6b-v2`, `engine="nemo"`, `nemo_model=...`); `ENGINES["nemo"]=NeMoEngine`; `_MODEL_VRAM_COST`; optional `nemo:<model>` ad-hoc regex.
6. **Thread config** — `RunConfig.nemo_python`, `--nemo-python` flag, `cfg=RunConfig(...)` wiring, `_config_to_dict`, `main()` pre-flight check.
7. **Tests** — update `test_engine_contract.py:27` to `{'faster-whisper','nim','whisperx','nemo'}`; add `FakeNeMoAdapter`-driven engine tests; `test_packaging.py` entry for `nemo_runner.py`; keep core torch-free.
8. **Live-validate on 5090** — confirm cu128 wheels + `cuda True`; measure peak VRAM (NVML) for both models; run Canary-Qwen chunked vs Whisper Large-V3-Turbo on the existing 12-lecture corpus; sanity-check chunk-boundary WER and hallucination signals.
9. **Docs + decision log** — record the validated triple, the captions decision, and "NeMo added in-file; split deferred to a standalone PR" in CLAUDE.md **and** AGENTS.md mirror (per the cross-agent sync rule), plus SPEC.md.
10. **(Separate later PR)** engines/ package split with re-export shim, gated on the full 231-test suite + CLI/pip smoke.

## Open questions to resolve in brainstorming

1. **Captions for Canary-Qwen:** WER-only/no-VTT, or forced-align its text? If aligning, reuse the WhisperX align stage or NeMo's own aligner — which gives better cue timing with least new dependency?
2. **One torch venv or two?** Standardize a shared `.venv-asr-torch` for WhisperX + NeMo, or keep `.venv-whisperx` / `.venv-nemo` separate? Hinges on whether both can agree on one torch/CUDA pin (findings lean separate, due to NeMo's pins).
3. **Exact version triple** on Windows 11 / RTX 5090: PyPI `nemo_toolkit` 2.7.3 vs git-trunk + which torch+cu128? Needs a real install test. And does any dep force WSL/Linux despite the model card listing Windows?
4. **Lighter first target?** Is classic Canary (`nemo.collections.asr` EncDecMultiTaskModel, e.g. canary-1b) or Parakeet-v2 a lower-risk *first* NeMo engine than SALM/Canary-Qwen (which pulls a Qwen LLM into the stack)? Ship Parakeet-v2 first, Canary-Qwen second?
5. **Does chunked (40s non-overlapping) inference materially raise Canary-Qwen WER** on continuous lecture speech vs its 5.63% short-clip leaderboard number? Only a real-corpus run answers this.
6. **Audio decode:** does NeMo accept arbitrary audio files directly (like `whisperx.load_audio`), or does it need 16kHz mono PCM like the NIM path (`decode_to_pcm16` at `:920`)? Determines whether `nemo_runner.py` needs its own decode step.
7. **Ad-hoc `nemo:<model>` ids** (mirroring `nim:<name>`), or registered-MODELS-only for v0.4?
8. **Diarization/DER for NeMo?** Canary is ASR-only, but NeMo has Parakeet/Sortformer diarization — does `NeMoResult` need the `speaker_segments`/`der` fields `ClipResult` already supports, or is v0.4 transcription-only?
9. **Wait for the June 2026 post-split NeMo Speech release** before pinning (extras names/pins may shift), or pin now against 2.7.3?
10. **Does v0.4 ship pip-install + PyInstaller binaries simultaneously with NeMo** (SPEC.md:153)? If yes, that's another reason to keep the package split out of this release.
11. **Reuse `NimEngine` for a NeMo NIM/HTTP path** instead of a new family (SPEC.md:86 notes NeMo often deploys via NIM)? Would change whether NeMo is truly a "4th family" and whether the split gate is met — but conflicts with "local in-venv default."
